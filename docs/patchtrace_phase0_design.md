## Problem Statement

AI coding agents can produce patches faster than developers can comfortably verify them. After a session, the developer has a requested task, agent claims, a changed working tree, a diff, test or command output, and uncertainty about what was actually verified.

PatchTrace helps developers verify agent work against the requested task and local evidence before accepting the patch.

PatchTrace is not a generic AI code reviewer. It is not a correctness oracle. It does not try to prove code is safe. It compares the task, the agent's stated work, the actual patch, and the available evidence, then produces a conservative verification brief for human review.

## What Makes This Cool

The cool part is the reversal of the usual AI-code-review framing. Most tools try to answer: "are there bugs in this PR?" PatchTrace answers: "did the agent actually do what it was asked to do, and where does the evidence fail to support the agent's claims?"

The demo moment should be concrete:

```text
Agent claim:
Added duplicate Stripe webhook protection and tests.

PatchTrace verdict:
Partially supported. The patch touches webhook and entitlement logic, and a billing webhook test exists, but the provided test evidence does not prove duplicate-event behavior. Review these files first. Production Stripe settings cannot be verified from local materials.
```

That moment shows claim skepticism, test-quality judgment, risk awareness, and conservative boundaries without claiming automated correctness.

## Constraints

- V0 is local-first.
- V0 is CLI-first.
- Markdown is the core output.
- JSON can be produced as a structured secondary output.
- HTML report is later, not Phase 0/V0.
- No SaaS, auth, teams, GitHub OAuth, cloud sync, public proof, social output, video replay, or correctness scoring in V0.
- AI/LLM usage is optional and should not be required for the first strong version.
- Project docs should describe the product directly and avoid meta framing about why it is being built.
- The current project directory is not yet a Git repository; Phase 1 foundation should create or move into a real repo before any git-diff-based implementation work.

## Premises

1. PatchTrace verifies agent work, not general code correctness.
2. The primary comparison is: requested task -> agent claims -> actual patch -> test/command evidence.
3. V0 should be fixture-first: hand-written ideal briefs before a clever analyzer.
4. V0 should be local-first CLI, with Markdown as the core output.
5. The long-term direction is PR and agent workflow integration, but not in V0.
6. Project docs should describe the product directly and avoid meta framing about why it is being built.

## Landscape Awareness

Search results confirm a crowded direction around AI-assisted code review, PR comment generation, and bug finding. That is not the best first category for PatchTrace.

Useful external signals:

- GitHub Copilot cloud agent docs frame agent-created PRs as a workflow that organizations need to measure through PR outcomes, merges, and time to merge: https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-cloud-agent
- SWE-PRBench reports that frontier models detected only 15-31% of human-flagged PR review issues in a diff-only setup: https://arxiv.org/abs/2603.26130
- Automated Code Review In Practice found AI review useful, but also associated with faulty, unnecessary, or irrelevant comments and longer PR closure time: https://arxiv.org/abs/2412.18531
- Rethinking Code Review in the Age of AI argues for human-controlled quality gates in agentic review workflows: https://arxiv.org/abs/2605.17548
- Early-Stage Prediction of Review Effort in AI-Generated Pull Requests frames agent PRs as a review-attention problem and suggests early triage using structural cues: https://arxiv.org/abs/2601.00753

Conclusion: PatchTrace should avoid competing as "another AI reviewer." Its sharper position is an agent-work verification layer: task, claims, patch evidence, test evidence, mismatch map, and review-first brief.

## Approaches Considered

### Approach A: Fixture-First Verification Brief Compiler

Summary: Create 3-5 realistic fixtures and hand-write ideal `VERIFICATION_BRIEF.md` outputs first. Then build a small CLI that uses simple rules to generate similar briefs from git diff, changed files, agent summary, and test output.

Effort: S/M
Risk: Low

Pros:
- Fastest path to a tool that can be dogfooded after real agent sessions.
- Forces a good judgment model before implementation complexity.
- Makes tests/evals natural from the first implementation slice.

Cons:
- Less magical in the first demo.
- Early analysis will be intentionally shallow.
- Requires discipline not to jump too early into LLMs, dashboards, or PR automation.

Reuses:
- Starting brief scenarios.
- Claim/evidence support model.
- Risk rules.
- Test-quality rules.
- Markdown verification brief structure.

### Approach B: Explicit Domain Engine From Day One

Summary: Build the full TypeScript domain pipeline immediately: collect material, parse diff, extract claims, classify risk, assess tests, rank review-first files, and build a verdict.

Effort: M/L
Risk: Medium

Pros:
- Strong long-term architecture.
- Clear path to JSON, HTML, and future PR integrations.
- Looks and behaves like a serious devtool internally.

Cons:
- Slower to dogfood.
- Higher over-engineering risk before proving the output is useful.
- Easy to build a clean pipeline that still produces an average report.

Reuses:
- Suggested architecture from the project brief.
- TypeScript schemas.
- Rules modules.

### Approach C: PR-Ready Report Prototype

Summary: Keep the CLI local, but shape the first Markdown output as a PR-ready artifact with sections that can later be pasted into a PR description or review comment.

Effort: M
Risk: Medium

Pros:
- Aligns with the 10x direction of PR and agent workflow integration.
- Makes the demo easier for technical reviewers to understand.
- Tests the eventual integration format early.

Cons:
- Can distract from local usefulness.
- Risks premature GitHub-specific decisions.
- Weaker as a pure verification engine if presentation becomes the focus.

Reuses:
- Markdown brief.
- Review-first list.
- Cannot-verify discipline.
- Future PR integration direction.

## Recommended Approach

Choose Approach A, with one constraint borrowed from Approach C: the Markdown report should be PR-friendly, but V0 should not build a PR integration.

The first useful version should be a local fixture-first verification brief compiler. It should make a developer faster at reviewing agent output without pretending to find every bug or prove correctness.

## Success Criteria

Phase 0 is done when:

- The product category is clear: agent-work verification, not AI code review.
- V0 scope is clear: local CLI, fixture-first, Markdown core output.
- The first 3-5 fixtures are selected.
- The ideal `VERIFICATION_BRIEF.md` shape is accepted.
- Non-goals are explicit enough to block scope creep.

V0 is useful when:

- A developer can generate a useful verification brief in under 5 minutes from local patch material and pasted agent/test notes.
- The report identifies unsupported or weakly supported agent claims.
- The report makes weak/missing test evidence visible.
- The report produces file-specific review-first guidance.
- The report includes cannot-verify items without creating false confidence.
- The developer would voluntarily use it after another AI-agent coding session.

## Distribution Plan

V0 distribution should be local CLI only.

Recommended initial path:

- Package as a local Node/TypeScript CLI during development.
- Expose `patchtrace analyze` as the first real command.
- Publish later through npm or GitHub Releases only after dogfood proves repeated usefulness.
- CI should run typecheck, lint, tests, and fixture/eval checks before any release.

No GitHub App, GitHub OAuth, hosted service, team workspace, or SaaS distribution in V0.
