## Important Distinctions

| Concept A | Concept B | Difference | Why it matters |
|---|---|---|---|
| Session recording | Patch analysis | Session recording captures what the agent said and did in the terminal. Patch analysis reads changed files and diffs. | Claim-vs-evidence needs both. |
| Patch verification | AI code review | Patch verification checks whether agent claims are backed by local evidence. AI code review tries to find bugs or judge code quality broadly. | PatchTrace should stay evidence-backed and avoid generic reviewer noise. |
| Claim support | Code correctness | Claim support says whether evidence supports what the agent said. It does not prove the code is correct. | Prevents false confidence. |
| Tests passed | Test evidence is strong | Passing tests may be unrelated, shallow, or too mocked. Strong test evidence checks the risky behavior. | Keeps test-quality review useful. |
| Cannot verify | Disclaimer | Cannot-verify items are actionable evidence gaps, not legal boilerplate. | Helps developers know what still needs manual/provider/runtime proof. |
| Review first | Warning list | Review-first output names specific files and reasons. Warning lists can be vague and unactionable. | Keeps reports short and useful. |
| Summary | Verification brief | Summary is the quick next-action artifact. Verification brief is the detailed evidence record. | The product must support fast action and deeper review. |
| Agent feedback | Verification brief | Agent feedback is written for the coding agent to continue work. Verification brief is written for the human reviewer. | This is the new Python V0 wedge. |
| Patch-only analysis | Full analysis | Patch-only analysis lacks session claims/transcript evidence. Full analysis has transcript plus patch material. | Prevents false confidence when watcher/manual mode lacks session data. |
| Local-first | Cloud/SaaS | Local-first keeps code, diffs, and transcripts on the developer machine by default. | Protects privacy and keeps V0 simple. |

## Naming Conventions

- User-facing copy uses: PatchTrace, Codex session, session transcript, agent
  claim, patch evidence, session evidence, test evidence, cannot verify, review
  first, summary, agent feedback, verification brief, conservative verdict.
- CLI/package naming uses: `patchtrace`.
- Python package naming uses: `src/patchtrace/`.
- Code package naming uses capability packages: `cli`, `session`, `adapters`,
  `vcs`, `analysis`, `reports`, `models`, and `storage`.
- Run folder path uses: `.patchtrace/runs/<run-id>/`.
- Required report filenames use: `SUMMARY.md`, `AGENT_FEEDBACK.md`,
  `VERIFICATION_BRIEF.md`.
- Billing/payment terms use: N/A for V0 because PatchTrace has no billing or
  paid access.
- Workspace/tenant terms use: N/A for V0 because PatchTrace has no accounts,
  workspaces, or tenant model.
